# 项目系统设计精炼文档

> 项目名称：AI 客服质检系统 / SmartQA Service Cloud  
> 当前版本：P0  
> 设计日期：2026-06-25  
> 设计原则：以 `项目需求精炼总结.md` 为最高优先级，其他 docs 只作为背景参考。

## 1. 设计边界

P0 只做千牛聊天记录质检主链路：

```text
千牛源数据
  -> ODS 原始入库
  -> 客服/客户/店铺/商品/会话/消息标准化
  -> 规则与 Prompt 拼接
  -> AI 质检
  -> 扣分问题、证据消息、客服表现统计
  -> 老板端和客服端展示
```

P0 明确不做：

| 不做项 | 处理方式 |
|---|---|
| 客服申诉 | 系统不出现申诉提交、处理、状态流转页面和表 |
| 微信聊天质检 | 先发微信只做 P1 转化客户打通预留 |
| 订单/售后/ROI/生意参谋 | 当前无完整数据支撑，不展示伪指标 |
| AI 生图分析 | 不进入 P0 系统菜单 |
| 复杂权限 | 只做老板全局权限和客服本人数据权限 |
| 复杂 AI 工作流 | 不做多模型投票、RAG、人工复核流转 |

## 2. 技术架构

系统基于 FastApiAdmin 二开，遵循最小使用原则。

| 层级 | 方案 |
|---|---|
| 前端 | 复用 FastApiAdmin Vue3 + TypeScript + Element Plus 后台 |
| 后端 | 复用 FastApiAdmin FastAPI + SQLAlchemy + Pydantic |
| 数据库 | MySQL 8，存放系统库、ODS、DWD、QC、统计表 |
| 缓存/会话 | Redis，复用框架能力 |
| 定时任务 | 复用 FastApiAdmin `module_task` / APScheduler |
| AI 调用 | 阿里/Qwen 类模型配置从环境变量或加密参数读取 |
| 权限 | 复用 FastApiAdmin 用户、角色、菜单、按钮权限 |
| 日志 | 复用操作日志，新增模型调用日志 |

二开目录建议：

```text
backend/app/plugin/module_smartqa/
  qianniu_data/        千牛数据源、批次、同步
  dimension/           店铺、商品、客服、客户
  conversation/        会话和消息
  qc_rule/             质检规则和 Prompt 模板
  qc_task/             质检任务执行
  qc_result/           质检结果、问题、证据
  dashboard/           老板端与客服端统计接口

frontend/web/src/views/module_smartqa/
  dashboard/
  qianniu-data/
  conversations/
  qc-tasks/
  qc-results/
  staff-performance/
  qc-rules/
  staff-users/
```

框架功能取舍：

| FastApiAdmin 能力 | P0 使用方式 |
|---|---|
| 登录认证 | 直接复用 |
| 用户/角色/菜单/按钮权限 | 直接复用 |
| 部门/岗位 | 可保留但不作为核心业务依赖 |
| 字典/配置 | 用于状态、风险等级、模型配置别名 |
| 操作日志 | 直接复用 |
| 定时任务 | 用于同步、扩表、质检、统计刷新 |
| 文件管理 | 后续 Excel 导入可用，P0 可隐藏菜单 |
| 多租户 | 单公司先隐藏菜单，不作为 P0 业务概念 |
| 代码生成 | 开发期可用，生产菜单隐藏 |
| 示例、发票、订单、套餐、内置 AI 聊天 | 不进入 P0 业务菜单 |

## 3. 角色与权限

系统最高权限是老板端。技术超管只作为部署维护账号存在，不作为业务角色。

| 角色 | 数据范围 | 功能范围 |
|---|---|---|
| 老板 | 全部店铺、全部客服、全部客户、全部会话 | 创建客服账号、绑定客服、查看总览、执行质检、看结果、配置规则 |
| 客服 | 仅自己负责的会话、质检结果和建议 | 查看我的会话、我的质检、我的改进建议 |

权限落地规则：

1. 老板账号拥有 `is_superuser` 或老板角色，业务数据不加客服过滤。
2. 客服账号必须绑定 `dim_staff.sys_user_id`。
3. 客服端所有列表接口默认加过滤：

```text
dwd_qn_conversation.staff_id = 当前登录用户绑定的 dim_staff.staff_id
```

4. 菜单权限控制功能可见，接口权限控制按钮动作，数据权限控制查询范围。
5. 客服不能查看其他客服的客户列表、会话详情、质检详情和排名，除非后续老板授权。

## 4. 客服账号初始化

当前已有客服需要先创建好，初始化流程如下：

1. 从 `ods_qn_shop_record.seller_wangwang` 去重生成 `dim_staff`。
2. 从 `shop_name + seller_wangwang` 生成 `dim_staff_account`。
3. 解析客服展示名：优先取 `seller_wangwang` 冒号后的名称，例如 `店铺:思琴` 取 `思琴`。
4. 为每个客服创建 `sys_user` 登录账号，并写回 `dim_staff.sys_user_id`。
5. 初始密码采用随机密码或统一临时密码，首次登录强制修改。
6. 如果后续新增 `seller_wangwang`，同步任务创建未绑定客服记录，由老板确认后创建系统账号。

建议不要把千牛原始账号直接当登录名强依赖。登录名可使用客服编号或短账号，千牛账号保存在 `dim_staff_account.account_name`。

## 5. 数据处理链路

```text
千牛源库 qn_chat_record / qn_shop_record
  -> ods_import_batch
  -> ods_qn_chat_record / ods_qn_shop_record
  -> dim_shop / dim_product / dim_staff / dim_staff_account / dim_customer / dim_customer_identity
  -> dwd_qn_conversation / dwd_qn_message / dwd_customer_staff_relation
  -> qc_task
  -> 模型调用
  -> qc_result / qc_issue / qc_issue_evidence / model_call_log
  -> 老板端统计 / 客服端统计
```

会话业务键：

```text
conversation_key = source_system + "|" + relation_id + "|" + business_id
```

客户识别规则：

1. `seller_wangwang` 是客服账号。
2. 同会话内 `chat_target != seller_wangwang` 的账号是客户真实淘宝/旺旺账号。
3. `buyer_wangwang` 是脱敏展示值，只能保存为别名，不能作为客户唯一键。

消息说话方规则：

| 条件 | speaker_type |
|---|---|
| `chat_target = seller_wangwang` | staff |
| `chat_target = customer_account` | customer |
| 无法匹配 | unknown |

## 7. 去重与增量同步设计

千牛数据会持续更新，所以同步必须幂等。重复跑同一批数据不应产生重复消息、重复会话或重复质检结果。

### 7.1 批次级幂等

1. 每次同步创建 `ods_import_batch`。
2. 所有 ODS 表写入都采用 upsert。
3. 每条 ODS 记录保存：

```text
source_id
row_hash
first_seen_batch_id
last_seen_batch_id
created_at
updated_at
```

4. `row_hash` 不变则跳过业务重算。
5. `row_hash` 变化则更新 ODS，并触发对应会话重算和质检失效。

### 7.2 聊天明细去重

优先使用源表 `id`：

```text
unique_key = source_system + "qn_chat_record" + id
```

如果后续发现源表 `id` 不稳定，使用兜底指纹：

```text
message_fingerprint = sha256(
  relation_id + "|" +
  business_id + "|" +
  chat_time + "|" +
  chat_target + "|" +
  normalized(chat_content)
)
```

同一指纹再次出现时只更新 `last_seen_batch_id`，不新增消息。

### 7.3 咨询业务记录去重

`qn_shop_record` 以 `relation_id + business_id` 为稳定业务键：

```text
unique_key = source_system + relation_id + business_id
```

该表的 `status`、`end_time`、`product_name` 可能后续变化，所以必须允许更新。更新后同步刷新 `dwd_qn_conversation.qn_status`、时间范围、商品字段和 `data_hash`。

### 7.4 会话与质检版本

会话数据版本：

```text
conversation_data_hash = sha256(
  conversation_key +
  qn_status +
  staff_id +
  customer_id +
  product_key +
  ordered(message_id + message_hash)
)
```

质检任务版本判断：

| 情况 | 处理 |
|---|---|
| 会话数据未变，规则未变 | 不重复创建任务 |
| 会话数据变化 | 原任务标记 `stale`，创建新任务 |
| 规则或 Prompt 版本变化 | 创建新任务，历史结果保留 |
| 模型调用失败 | 任务 `failed`，允许手动重试 |

`qc_task` 建议唯一约束：

```text
conversation_id + conversation_data_hash + rule_version + prompt_version
```

### 7.5 同步水位

推荐同步策略：

| 表 | 策略 |
|---|---|
| `qn_chat_record` | 按 `create_time + id` 增量，保留 30 到 120 分钟重叠窗口 |
| `qn_shop_record` | 滚动重拉最近 7 天，upsert 状态和结束时间 |

原因：聊天明细基本追加，咨询业务状态可能从“待定”变为“落实下单/付款”，必须滚动修正。

## 8. AI 质检设计

质检输入按会话组织，消息必须携带可追溯 ID：

```text
[msg_1231][2026-06-23 23:11:09][staff][芊姿彩润凡专卖店:思琴] 欢迎您光临本店
[msg_1235][2026-06-23 23:11:23][customer][aizgwp] 陶粒一般买多少呢
```

P0 默认规则维度：

| 维度 | 说明 |
|---|---|
| 服务态度 | 冷漠、敷衍、不耐烦、攻击性表达 |
| 响应及时 | 客户关键问题后长时间未回复或连续忽略 |
| 问题解决 | 未正面回答规格、价格、定制、物流等问题 |
| 合规风险 | 违规承诺、平台外交易、不当话术 |
| 成交推进 | 高意向客户未确认需求、未报价、未引导下单 |

AI 输出硬性要求：

1. 只能输出合法 JSON。
2. 每个 `issue` 必须包含 `evidence_message_ids`。
3. `evidence_message_ids` 必须存在于 `dwd_qn_message`。
4. 无证据的问题不得正式扣分。
5. 原始模型响应必须保存到 `model_call_log`。
6. JSON 解析失败时自动重试一次，仍失败则任务失败并保留原文。

结果等级：

| 分数 | result_level | 说明 |
|---:|---|---|
| 90-100 | pass | 基本合格 |
| 70-89 | warning | 有改进项 |
| 0-69 | fail | 明显不合格 |

风险等级使用：

```text
none / low / medium / high / critical
```

## 11. 定时任务

| 任务 | 建议频率 | 说明 |
|---|---|---|
| `sync_qianniu_ods` | 每 10 到 30 分钟 | 从千牛源库同步 ODS |
| `build_qianniu_dwd` | 同步后触发 | 扩表生成维表、会话、消息 |
| `create_qc_tasks` | 同步后触发或每日定时 | 为待质检会话创建任务 |
| `run_qc_tasks` | 按队列执行 | 调用模型并入库结果 |
| `refresh_qc_stats` | 每小时或任务后触发 | 刷新老板端和客服端统计 |
| `seed_staff_users` | 首次上线手动执行 | 创建已有客服账号 |

任务并发控制：

1. 同一时间只允许一个千牛同步任务运行。
2. 同一会话同一版本只允许一个质检任务运行。
3. 模型调用设置并发上限，避免费用和限流失控。

## 13. 部署设计

P0 最小部署：

```text
Nginx
  -> frontend/web 静态资源
  -> backend FastAPI
      -> MySQL
      -> Redis
      -> 千牛源库只读连接
      -> 阿里模型服务
```

部署注意：

1. 千牛源库账号建议只读权限。
2. 业务库和千牛源库分开，避免同步逻辑误写源库。
3. 定时任务和 API 服务可先同进程部署，数据量扩大后再拆任务 worker。
4. 每日备份 MySQL，至少保留 7 到 30 天。
5. `model_call_log` 可能增长较快，后续按月归档。

## 14. P0 验收标准

| 验收项 | 标准 |
|---|---|
| 源表入库 | 能导入 13,756 条聊天明细和 598 条业务记录 |
| 会话匹配 | 598 个 `relation_id + business_id` 会话全部可匹配 |
| 客服初始化 | 能从 18 个 `seller_wangwang` 生成客服并创建登录账号 |
| 客户识别 | 客户使用非客服 `chat_target`，不使用脱敏 `buyer_wangwang` 做唯一键 |
| 消息展示 | 每个会话能按时间展示完整聊天记录 |
| 说话方识别 | 能正确标记 `staff/customer/unknown` |
| 重复同步 | 同一批数据重复导入不会重复生成消息、会话、任务 |
| 状态更新 | `qn_shop_record.status` 更新后会话状态能同步刷新 |
| 质检执行 | AI 输出合法 JSON 并入库 |
| 证据追溯 | 每个扣分项能定位到 `dwd_qn_message.message_id` |
| 权限隔离 | 老板看全部，客服只看自己的会话和结果 |
| 页面范围 | 系统不出现客服申诉、订单、售后、ROI、生意参谋页面 |

## 15. 后续 P1 预留

P1 只预留先发微信“转化客户”打通，不把微信作为 P0 质检主链路。

建议新增：

| 表 | 用途 |
|---|---|
| `ods_xianfa_converted_customer` | 先发微信转化客户原始表 |
| `map_qn_xianfa_customer` | 千牛客户与先发客户映射 |

匹配优先级：

1. 淘宝号/旺旺号完全一致，自动匹配。
2. 手机号一致，高置信匹配。
3. 昵称相似但无强身份，不自动合并，进入人工确认。
