# 项目需求精炼总结

> 项目名称：AI 客服质检系统 / SmartQA Service Cloud  
> 当前版本：P0，基于千牛近两天真实聊天记录做 AI 质检  
> 数据样本：`ai_qc_app/storage/qianniu_exports/qianniu_export_20260625_103316.xlsx`  
> 文档目的：基于真实千牛数据，明确 P0 需求范围、扩表方案、核心规则和示例数据

## 1. 项目定位

第一版只做千牛聊天记录质检：

```text
千牛聊天记录
    -> 数据清洗与扩表
    -> 客服/客户/店铺/商品/会话/消息标准化
    -> 质检规则 + Prompt 拼接
    -> AI 质检
    -> 扣分问题、证据消息、客服表现统计
```

当前不做客服申诉，不做订单、售后、ROI、生意参谋、生图系统分析。后续只接入先发微信侧“转化客户”，通过淘宝号/旺旺号与千牛客户打通，不把微信作为第一版质检主链路。

## 3. P0 核心需求

| 目标 | 说明 |
|---|---|
| 千牛数据入库 | 将两张源表增量导入 ODS 原始层 |
| 自动扩表 | 从增长源表中沉淀客服、客户、店铺、商品、会话、消息表 |
| 会话质检 | 以 `relation_id + business_id` 生成的会话为 AI 质检对象 |
| 规则配置 | 建立质检规则表，用于 Prompt 拼接和版本追踪 |
| 证据追溯 | 每个扣分项必须关联到 `dwd_qn_message.message_id` |
| 结果统计 | 按客服、店铺、商品、状态、问题类型统计质检结果 |
| 先发预留 | 后续通过淘宝号/旺旺号关联微信转化客户 |

## 4. P0 不做内容

| 不做项 | 说明 |
|---|---|
| 客服申诉 | 第一版不做申诉提交、处理、状态流转 |
| 微信聊天质检 | 先发微信只用于后续转化客户打通 |
| 订单/售后/ROI | 当前数据没有订单金额、退款、售后责任等字段，不展示伪指标 |
| 复杂权限 | 第一版只区分老板/管理员和客服本人数据 |
| 复杂 AI 能力 | 不做多模型投票、RAG、复杂工作流 |

## 6. 关键生成规则

### 6.1 会话生成

```text
conversation_key = relation_id + "|" + business_id
conversation_id = hash(source_system + relation_id + business_id)
```

时间字段：

1. `start_time` 优先取 `qn_shop_record.start_time`。
2. `end_time` 优先取 `qn_shop_record.end_time`。
3. 如业务表时间缺失，则回退取同会话消息最早/最晚 `chat_time`。

### 6.2 客服识别

```text
staff_account = qn_shop_record.seller_wangwang
```

当前真实数据中，`seller_wangwang` 与聊天明细里的客服 `chat_target` 可以匹配。

### 6.3 客户识别

```text
同一会话的 chat_target 列表
    - seller_wangwang 对应客服
    - 非客服 chat_target 对应客户真实淘宝/旺旺账号
```

`buyer_wangwang` 只保存为脱敏展示别名，不作为唯一身份。

### 6.4 消息说话方

| 条件 | `speaker_type` |
|---|---|
| `chat_target = staff_account` | `staff` |
| `chat_target = customer_account` | `customer` |
| 无法匹配 | `unknown` |

### 6.5 消息去重

优先使用源表 `id`。如果后续源库 `id` 不稳定，则使用：

```text
hash(relation_id + business_id + chat_time + chat_target + chat_content)
```

## 9. AI 输出结构要求

AI 必须输出合法 JSON，不能输出 Markdown 或解释性文本。

```json
{
  "conversation_id": "conv_1782290881765",
  "score": 86,
  "result_level": "pass | warning | fail",
  "risk_level": "none | low | medium | high | critical",
  "summary": "会话总结",
  "dimension_scores": {
    "attitude": 95,
    "response": 90,
    "solution": 85,
    "compliance": 100,
    "conversion": 80
  },
  "issues": [
    {
      "rule_code": "rule_conversion",
      "severity": "low",
      "title": "成交推进可加强",
      "reason": "判断原因",
      "evidence_message_ids": ["msg_1315", "msg_1317"],
      "suggested_action": "处理建议",
      "suggested_reply": "推荐回复"
    }
  ],
  "confidence": 0.9
}
```

硬性要求：

1. 每个 `issue` 必须有 `evidence_message_ids`。
2. 证据 ID 必须存在于 `dwd_qn_message`。
3. 无证据的问题不得正式扣分。
4. 原始模型响应必须保存到 `model_call_log` 或 `qc_task.response_json`。

## 14. 总结

基于真实千牛数据，第一版应围绕以下主链路实施：

```text
qn_shop_record + qn_chat_record
    -> ODS 原始层
    -> dim_staff / dim_customer / dim_shop / dim_product
    -> dwd_qn_conversation / dwd_qn_message
    -> qc_rule / qc_prompt_template / qc_task
    -> qc_result / qc_issue / qc_issue_evidence
    -> 客服表现和问题统计
```

最关键的设计点是：客户不要用脱敏的 `buyer_wangwang` 做唯一键，而要从聊天明细 `chat_target` 中提取真实客户账号；扣分不要只存 AI 结论，而必须落到可追溯的消息证据。
