# 老板端客服端功能分析文档

> 项目名称：SmartQA Service Cloud / AI 客服质检系统
> 当前版本：P0
> 生成日期：2026-06-26
> 最高优先级依据：`docs/项目需求精炼总结.md`
> 补充参考：`docs/项目系统设计精炼文档.md`、当前 SmartQA 代码结构

## 1. 结论先行

SmartQA 第一版不是通用后台，也不是泛泛的客服管理系统。P0 只围绕一条主链路做真实可用功能：

```text
千牛聊天记录
  -> 真实源库同步
  -> ODS 原始入库
  -> 客服/客户/店铺/商品/会话/消息标准化
  -> 规则 + Prompt
  -> 阿里 Qwen AI 质检
  -> 扣分问题、证据消息、建议话术、客服表现统计
  -> 老板端管理闭环 / 客服端个人改进闭环
```

角色定位必须清楚：

| 角色 | 本质目标 | 数据范围 | 功能关键词 |
|---|---|---|---|
| 老板端 | 看全局、跑质检、抓问题、管规则、管客服账号 | 全部店铺、全部客服、全部客户、全部会话、全部质检结果 | 经营视角、质检管理、追溯证据、人员表现、规则配置 |
| 客服端 | 看自己、改自己、按证据改话术 | 仅当前登录客服绑定的 `dim_staff` 数据 | 我的会话、我的质检、我的问题、我的建议 |

第一版明确不做：

| 不做项 | 处理方式 |
|---|---|
| 客服申诉 | 不出现申诉提交、申诉处理、申诉状态流转页面和表 |
| 微信聊天质检 | 先发微信只作为 P1 转化客户打通预留 |
| 订单、售后、ROI、生意参谋 | 当前千牛聊天数据无可靠字段支撑，不展示伪指标 |
| AI 生图分析 | 不进入 P0 菜单 |
| 复杂权限流 | 只做老板全局权限和客服本人数据权限 |
| 假数据看板 | 页面指标必须来自真实同步、真实扩表、真实质检结果 |

## 2. 功能完成口径

后续不能把“页面搭出来”当成功能完成。SmartQA 的任一功能都必须同时满足下面 5 条：

| 口径 | 必须满足 |
|---|---|
| 页面 | 有符合角色的页面、筛选、列表、详情、操作状态 |
| API | 前端调用真实后端接口，不写死 mock |
| 数据 | 数据来自 `smartqa` 业务库，源头可追溯到千牛源库 |
| 权限 | 老板看全量，客服只看自己，直接请求接口也不能越权 |
| 验收 | 能用真实千牛数据和真实阿里模型跑通一轮 |

文档里的样本数据曾记录为 13,756 条聊天、598 条会话；用户最新观察源库约为 13,543 条聊天、555 条会话。由于源库会持续变化，验收时不要把样本行数写死为唯一标准，而应以当次源库查询快照为准，并在千牛数据源页显示源表行数、入库行数、会话数和最新聊天时间。

## 3. 老板端功能

老板端是系统最高业务角色。老板端不是只看报表，还要能完成“同步数据 -> 创建任务 -> 执行质检 -> 查看问题 -> 定位证据 -> 管理客服 -> 调整规则”的闭环。

### 3.1 老板工作台总览

目标：老板登录后第一眼知道整体服务质量。

| 功能 | 说明 |
|---|---|
| 核心指标 | 会话数、已质检数、平均分、不合格数、高风险数、问题总数 |
| 问题分布 | 按态度、响应、问题解决、合规、成交推进统计 |
| 客服排行 | 按平均分、问题数、高风险数、质检量展示 |
| 店铺分布 | 按店铺统计会话数和质检结果 |
| 下钻 | 点击客服、店铺、问题类型能进入对应筛选后的结果列表 |

验收标准：

- 指标来自 `dwd_qn_conversation`、`qc_result`、`qc_issue`，不能是前端写死。
- 老板看到全量数据，不加客服本人过滤。
- 不展示成交金额、ROI、订单转化率、售后金额等无数据支撑指标。

### 3.2 千牛数据源

目标：老板能确认真实千牛数据有没有进系统，以及每次同步是否成功。

| 功能 | 说明 |
|---|---|
| 批次列表 | 展示 `ods_import_batch` 的批次号、开始时间、结束时间、状态、错误信息 |
| 源表统计 | 展示源库聊天行数、业务记录行数、会话数、最新聊天时间 |
| 入库统计 | 展示 ODS 行数、DWD 会话数、DWD 消息数 |
| 手动同步 | 支持从千牛源库同步到 ODS，并触发扩表 |
| 去重状态 | 重复同步不生成重复会话、重复消息、重复任务 |
| 失败追踪 | 同步失败要显示失败原因，不只显示失败两个字 |

验收标准：

- 同步必须读取真实千牛源库。
- `qn_chat_record` 按源 `id` 或消息指纹去重。
- `qn_shop_record` 按 `relation_id + business_id` upsert，允许状态变化刷新。
- 数据源页必须能解释“源库多少、入库多少、扩表多少、最新到什么时间”。

### 3.3 客服账号管理

目标：老板能把千牛客服身份变成系统客服账号，并维护登录权限。

| 功能 | 说明 |
|---|---|
| 客服列表 | 展示 18 个或最新源库识别出的客服、千牛账号、绑定系统账号、状态 |
| 初始化客服 | 从 `ods_qn_shop_record.seller_wangwang` 去重生成 `dim_staff` |
| 绑定账号 | 将 `dim_staff.sys_user_id` 绑定到系统登录用户 |
| 创建账号 | 老板可为未绑定客服创建登录账号 |
| 重置密码 | 老板可重置客服密码 |
| 启停账号 | 老板可启用、禁用客服账号 |
| 新客服提示 | 源库出现新的 `seller_wangwang` 后，提示老板确认创建 |

验收标准：

- 客服身份来自真实 `seller_wangwang`，不是手填假客服。
- 客服账号和千牛账号不要混为一谈，千牛账号保存在 `dim_staff_account.account_name`。
- 老板账号是业务最高权限，技术超管不作为日常业务角色。

### 3.4 会话管理

目标：老板能查看所有千牛会话，并从会话进入质检。

| 功能 | 说明 |
|---|---|
| 会话列表 | 展示会话 ID、店铺、客服、客户、商品、千牛状态、消息数、首响、质检状态 |
| 筛选 | 时间、店铺、客服、客户、商品、千牛状态、质检状态、关键词 |
| 会话详情 | 商品、买家、客服、开始/结束时间、完整聊天时间线 |
| 说话方 | 每条消息标记 `staff/customer/unknown` |
| 源数据追溯 | 保留 `relation_id`、`business_id`、源消息 ID |
| 创建质检 | 可对单个或批量会话创建 AI 质检任务 |

验收标准：

- 会话业务键是 `relation_id + business_id`。
- 客户唯一身份优先用同会话内非客服 `chat_target`，不能用脱敏 `buyer_wangwang` 做唯一键。
- 会话详情必须按时间完整展示消息，不能只展示摘要。

### 3.5 AI 质检任务

目标：老板能实际调用阿里模型完成批量质检，并知道失败原因。

| 功能 | 说明 |
|---|---|
| 任务列表 | 展示任务 ID、会话、规则版本、Prompt 版本、模型、状态、错误 |
| 创建任务 | 对待质检会话创建任务 |
| 批量执行 | 支持按选择或批次执行 |
| 失败重试 | 失败任务可重试，保留错误信息 |
| 过期任务 | 会话数据或规则变化后旧任务标记 `stale` |
| 模型日志 | 保存原始输入、输出、token、耗时、成功失败 |

验收标准：

- 模型使用阿里配置，默认模型按当前配置 `qwen3.7-plus`。
- AI 输出必须是合法 JSON。
- 每个扣分问题必须带 `evidence_message_ids`。
- 证据 ID 必须能在 `dwd_qn_message` 中找到。
- 无证据的问题不能正式扣分。

### 3.6 质检结果

目标：老板能从结果看到问题，并能追溯到原始聊天证据。

| 功能 | 说明 |
|---|---|
| 结果列表 | 分数、等级、风险、问题数、客服、店铺、商品、客户、时间 |
| 筛选 | 时间、店铺、客服、结果等级、风险等级、问题类型、关键词 |
| 结果详情 | 总分、维度分、问题列表、扣分、原因、建议动作、建议话术 |
| 证据消息 | 每个问题展示对应原始消息、说话方、时间 |
| 原始 JSON | 老板端可查看模型原始 JSON，便于排查 |
| 跳转会话 | 从质检详情能回到完整会话详情 |

验收标准：

- `qc_result`、`qc_issue`、`qc_issue_evidence` 三层数据完整。
- 结果统计与列表明细一致。
- 质检结果不是只显示分数，必须显示问题、证据和建议话术。

### 3.7 客服表现

目标：老板能判断哪个客服服务质量好、哪里有风险、应该培训什么。

| 功能 | 说明 |
|---|---|
| 客服排行 | 平均分、质检量、问题数、高风险数 |
| 客服详情 | 某个客服的会话列表、质检结果、问题分布 |
| 问题类型统计 | 态度、响应、问题解决、合规、成交推进 |
| 高风险聚合 | 查看高风险客服、高风险商品、高风险会话 |
| 时间对比 | 支持按日期范围看趋势 |

验收标准：

- 老板能从客服排行点进某个客服详情。
- 客服表现来自真实质检结果，不使用订单销售类伪指标。
- 该页面只给老板端，不给普通客服看全局排行。

### 3.8 质检规则配置

目标：老板能维护质检标准，且任务执行时使用可追溯版本。

| 功能 | 说明 |
|---|---|
| 规则列表 | 规则编码、名称、分类、扣分、严重程度、状态 |
| 规则编辑 | 编辑判断标准、扣分、严重程度 |
| Prompt 模板 | 维护千牛 P0 质检 Prompt 片段 |
| 版本发布 | 发布规则版本快照，任务引用固定版本 |
| 启停规则 | 支持启用、停用，不直接删除历史规则 |

验收标准：

- 规则必须参与 Prompt 拼接。
- `qc_task` 保存 `rule_version` 和 `prompt_version`。
- 历史结果保留当时使用的规则版本。
- 客服端不能看到规则配置入口。

## 4. 客服端功能

客服端不是缩小版老板端。客服端只服务一个目标：让客服知道自己哪句话、哪个会话、哪个问题需要改。

### 4.1 我的工作台

目标：客服登录后看到自己当前服务表现和待改进重点。

| 功能 | 说明 |
|---|---|
| 我的指标 | 我的会话数、我的已质检数、我的平均分、我的问题数、我的高风险数 |
| 我的待改进 | 按问题类型汇总自己最多的问题 |
| 高风险提醒 | 展示自己的高风险会话和原因 |
| 最近结果 | 展示最近完成质检的会话 |

验收标准：

- 所有数据默认加过滤：`dwd_qn_conversation.staff_id = 当前登录用户绑定的 dim_staff.staff_id`。
- 不展示全局客服排行。
- 不展示其他客服姓名和数据。

### 4.2 我的会话

目标：客服只能查看自己负责的千牛会话。

| 功能 | 说明 |
|---|---|
| 我的会话列表 | 展示自己负责的会话、客户、商品、状态、质检状态 |
| 筛选 | 时间、客户、商品、千牛状态、质检状态、关键词 |
| 我的会话详情 | 展示完整聊天时间线和说话方 |
| 关联结果 | 如果已质检，能跳转到对应质检结果 |

验收标准：

- 客服不能通过 URL 参数或接口请求查看其他客服会话。
- 客服会话列表不显示“客服筛选器”，因为范围天然就是自己。
- 默认不允许客服创建或执行质检任务；如后续开放，需要老板明确授权。

### 4.3 我的质检结果

目标：客服看到自己每个会话的质检结论、证据和建议话术。

| 功能 | 说明 |
|---|---|
| 我的结果列表 | 分数、等级、风险、问题数、商品、客户、时间 |
| 结果详情 | 维度分、问题、扣分、原因、建议动作、建议话术 |
| 证据定位 | 每个问题定位到原始消息 |
| 改进状态 | P0 可只展示，不做申诉或复杂整改流转 |

验收标准：

- 客服只能看自己的 `qc_result`。
- 必须显示证据消息，而不是只显示 AI 总结。
- 不出现申诉按钮、申诉状态、申诉处理入口。

### 4.4 我的改进建议

目标：把多个质检结果汇总成客服能执行的训练重点。

| 功能 | 说明 |
|---|---|
| 问题类型汇总 | 统计自己在态度、响应、解决、合规、成交推进上的问题数 |
| 高频问题 | 展示最近最常见的扣分标题和原因 |
| 推荐话术 | 汇总 AI 给出的建议话术 |
| 训练重点 | 给出本周或当前周期最应该改的 3 到 5 个点 |
| 跳转证据 | 从建议跳到具体质检结果和会话证据 |

验收标准：

- 建议必须从 `qc_issue.suggested_action`、`qc_issue.suggested_reply` 和证据消息汇总而来。
- 不能写死通用培训文案。
- P0 只做展示和跳转，不做复杂学习任务系统。

### 4.5 个人账号能力

目标：满足登录安全和基本账号使用。

| 功能 | 说明 |
|---|---|
| 修改密码 | 复用基础账号能力 |
| 查看个人信息 | 显示系统账号和绑定客服身份 |
| 登录态 | 复用框架登录与 token |

验收标准：

- 客服能用老板创建或初始化的账号登录。
- 如果后续启用首次登录改密，需要在登录后清晰引导。

## 5. 数据权限规则

权限必须在前端菜单、后端接口、数据库查询三层同时落地。

| 场景 | 老板端 | 客服端 |
|---|---|---|
| 工作台 | 全量统计 | 仅本人统计 |
| 千牛数据源 | 可见，可同步 | 不可见 |
| 客服账号管理 | 可见，可创建/绑定/重置 | 不可见 |
| 会话列表 | 全量，可按客服筛选 | 仅本人，不显示客服筛选 |
| 会话详情 | 全量可查 | 仅本人可查 |
| AI 质检任务 | 可创建、执行、重试 | 默认不可见 |
| 质检结果 | 全量，可筛选 | 仅本人 |
| 客服表现 | 全局排行和详情 | 不可见全局排行 |
| 规则配置 | 可见，可维护 | 不可见 |

后端硬性规则：

```text
老板账号：
  is_superuser = true
  或拥有 smartqa_boss 角色
  -> 不加 staff_id 过滤

客服账号：
  必须绑定 dim_staff.sys_user_id
  -> 所有会话/结果/证据/统计接口都加：
     dwd_qn_conversation.staff_id = 当前登录用户绑定的 dim_staff.staff_id
```

如果客服账号未绑定 `dim_staff`，所有客服业务数据接口应返回 403，并提示“当前账号未绑定客服身份”。

## 6. 菜单建议

### 6.1 老板端菜单

```text
SmartQA
  工作台总览
  千牛数据源
  会话管理
  AI 质检任务
  质检结果
  客服表现
  质检规则
  客服账号
```

### 6.2 客服端菜单

```text
SmartQA
  我的工作台
  我的会话
  我的质检结果
  我的改进建议
```

### 6.3 必须隐藏或删除的菜单

P0 生产菜单不应出现：

- 订单、发票、套餐、商品销售、库存、销售额等旧模板示例业务菜单。
- 客服申诉。
- ROI、生意参谋、售后金额、成交金额。
- 微信聊天质检。
- AI 生图分析。
- 代码生成等开发期工具菜单。

## 7. 当前实现差距

基于当前代码结构，SmartQA 已经有一些基础模块，但还不能认为项目功能完成。

### 7.1 已有基础

| 类型 | 当前已有 |
|---|---|
| 后端模块 | `qianniu_data`、`conversation`、`qc_task`、`qc_result`、`qc_rule`、`dashboard`、`staff_user` |
| 前端页面 | `dashboard`、`qianniu-data`、`conversations`、`qc-tasks`、`qc-results`、`staff-users` |
| 权限基础 | 后端已有老板识别和客服 `staff_id` 过滤辅助逻辑 |
| API 基础 | 已有会话、质检任务、质检结果、数据源、客服账号等接口 |

### 7.2 关键缺口

| 缺口 | 影响 | 后续处理 |
|---|---|---|
| 没有独立客服端页面 | 客服登录后仍像在看管理后台，不是“我的”工作流 | 新增 `my-dashboard`、`my-conversations`、`my-qc-results`、`my-improvements` |
| 前端缺少客服表现页 | 老板无法系统查看某个客服的问题分布和详情 | 新增 `staff-performance` 页面 |
| 前端缺少质检规则页 | 后端有 `qc_rule`，但老板无法在页面维护规则 | 新增 `qc-rules` 页面并接入规则/Prompt/版本接口 |
| 工作台角色不分 | 老板和客服应看到不同指标与操作 | 拆成老板工作台和客服我的工作台 |
| 筛选不完整 | 会话/结果无法按文档要求完整筛选 | 补时间、店铺、客服、客户、商品、状态、问题类型筛选 |
| 质检详情展示不够 | 需要完整展示维度分、建议动作、建议话术、证据消息 | 扩展结果详情 UI |
| 客服端不能出现全局排行 | 违反数据权限和产品定位 | 菜单和页面层隐藏，接口层继续强制过滤 |
| 默认示例后台菜单未彻底收敛 | 用户会误以为系统有订单、销售等不相关功能 | 生产菜单只保留 SmartQA 与必要系统账号能力 |
| 真实验收链路还需固化 | 不能停留在构建成功 | 用真实源库同步、真实扩表、真实阿里模型、真实页面检查验收 |

## 8. 开发优先级

### P0-1：先把角色和菜单立住

1. 清理生产菜单，只保留 SmartQA P0 业务菜单和必要系统菜单。
2. 老板角色显示老板端菜单。
3. 客服角色显示客服端菜单。
4. 直接访问无权限路由和接口必须失败。

### P0-2：补齐老板端闭环

1. 千牛数据源能展示真实同步状态。
2. 会话列表和详情按文档补全筛选与字段。
3. AI 质检任务能批量创建、执行、重试。
4. 质检结果详情补全维度分、问题、证据、建议话术。
5. 新增客服表现页。
6. 新增质检规则配置页。

### P0-3：补齐客服端闭环

1. 新增我的工作台。
2. 新增我的会话。
3. 新增我的质检结果。
4. 新增我的改进建议。
5. 确认客服账号只能看到自己绑定的 `dim_staff` 数据。

### P0-4：真实数据真实模型验收

1. 从千牛源库同步最新数据。
2. 扩表生成店铺、商品、客服、客户、会话、消息。
3. 用阿里模型执行全量或抽样质检。
4. 检查 `qc_result`、`qc_issue`、`qc_issue_evidence` 是否完整。
5. 用老板账号和客服账号分别登录页面验收。

## 9. 验收清单

### 9.1 老板端验收

| 验收项 | 标准 |
|---|---|
| 登录 | 老板账号能进入老板端菜单 |
| 工作台 | 显示真实会话数、质检数、平均分、问题数、高风险数 |
| 数据源 | 能看到最新源库同步批次、源表行数、会话数、最新聊天时间 |
| 客服账号 | 能看到从 `seller_wangwang` 生成的客服，并能绑定系统账号 |
| 会话 | 能按时间、店铺、客服、客户、商品、状态、质检状态筛选 |
| 会话详情 | 能看到完整聊天时间线和 `staff/customer` 说话方 |
| 质检任务 | 能创建任务、执行任务、查看失败原因、重试失败 |
| 质检结果 | 能查看分数、风险、问题、证据消息、建议话术 |
| 客服表现 | 能按客服查看平均分、问题数、高风险数和问题分布 |
| 规则配置 | 能维护规则、扣分、严重程度、Prompt 和版本 |
| 禁止项 | 不出现申诉、订单、售后、ROI、生意参谋、微信质检、生图分析页面 |

### 9.2 客服端验收

| 验收项 | 标准 |
|---|---|
| 登录 | 客服账号能进入客服端菜单 |
| 我的工作台 | 只显示自己的会话、质检、问题、高风险统计 |
| 我的会话 | 只显示自己 `staff_id` 对应会话 |
| 我的会话详情 | 只能打开自己的会话详情 |
| 我的质检结果 | 只显示自己的分数、问题、证据和建议话术 |

| 越权验证 | 修改 URL 或接口参数访问其他客服数据必须失败 |
| 禁止项 | 不显示全局客服排行、规则配置、客服账号管理、千牛数据源、AI 任务执行 |

### 9.3 数据一致性验收

| 验收项 | 标准 |
|---|---|
| 源表同步 | 当前源库聊天和会话数量能在数据源页展示 |

| 统计一致 | 工作台和客服表现统计能从明细结果反查一致 |

## 10. 最小可用完成定义

P0 真正完成时，应该能现场演示下面两条路径。

老板路径：

```text
老板登录
  -> 查看真实同步数据源
  -> 查看全量会话
  -> 批量创建并执行 AI 质检
  -> 查看质检结果详情
  -> 点击扣分问题看到证据消息
  -> 查看客服表现排行和某客服问题详情
  -> 调整质检规则并发布新版本
```

客服路径：

```text
客服登录
  -> 查看我的工作台
  -> 查看我的会话
  -> 打开我的质检结果
  -> 看到扣分原因、证据消息、建议话术
  -> 进入我的改进建议
  -> 明确自己下一步要改什么
```

只要上述任一路径还需要 mock 数据、假指标、绕过真实模型、或者客服能看到别人数据，就不能算 SmartQA P0 功能完成。
