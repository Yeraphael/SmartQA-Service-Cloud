# SmartQA Service Cloud 项目技术栈说明

本文档整理当前项目中实际使用或已配置的主要技术栈，并说明它们分别承担什么职责。项目整体是一个面向千牛客服会话数据的 AI 质检系统，采用前后端分离架构。

## 1. 整体架构

| 层级 | 技术/模块 | 用途 |
| --- | --- | --- |
| 前端 Web | Vue 3 + TypeScript + Vite | 构建后台管理页面、老板端/客服端工作台、质检任务与结果页面 |
| 后端 API | Python 3.12 + FastAPI | 提供登录认证、系统管理、SmartQA 业务接口、AI 质检接口 |
| 数据库 | MySQL 8.0 | 保存用户、权限、千牛同步数据、会话明细、质检任务和质检结果 |
| 缓存/会话 | Redis 7 | 保存登录会话、系统参数缓存、租户配置缓存、限流状态 |
| AI 模型 | 阿里百炼 qwen 系列 + OpenAI SDK | 通过 OpenAI 兼容接口调用阿里通义千问模型完成客服质检 |
| 定时任务 | APScheduler | 定时同步千牛源数据、定时抽样并执行每日 AI 质检 |
| 部署 | Docker Compose + Nginx | 编排 MySQL、Redis、后端、前端静态资源和反向代理 |

## 2. 后端技术栈

### 2.1 Web 框架与服务运行

| 技术 | 版本/位置 | 用途 |
| --- | --- | --- |
| Python | `>=3.12` | 后端运行语言 |
| FastAPI | `0.115.2` | Web API 框架，负责接口路由、依赖注入、请求/响应处理 |
| Uvicorn | `0.49.0` | ASGI 服务，用于运行 FastAPI 应用 |
| Typer | `0.26.7` | 命令行工具，提供 `python main.py run/revision/upgrade` 等命令 |
| Starlette Middleware | FastAPI 底层能力 | CORS、GZip、请求上下文、租户上下文等中间件基础 |

后端启动入口在 `backend/main.py`，核心应用创建函数是 `create_app()`。运行命令：

```bash
cd backend
python main.py run --env=dev
```

### 2.2 配置与环境变量

| 技术 | 用途 |
| --- | --- |
| pydantic-settings | 读取 `.env.dev`、`.env.prod` 等环境配置 |
| SettingsConfigDict | 控制环境变量大小写、额外字段处理、配置文件编码 |
| 项目配置类 `Settings` | 统一管理数据库、Redis、JWT、AI 模型、调度器、上传、Swagger 等配置 |

配置入口在 `backend/app/config/setting.py`。例如：

- `DATABASE_HOST` / `DATABASE_PORT` / `DATABASE_NAME`：业务数据库配置
- `REDIS_HOST` / `REDIS_PASSWORD`：Redis 配置
- `OPENAI_API_KEY` / `OPENAI_BASE_URL`：AI 模型调用配置
- `SMARTQA_ALI_MODEL_NAME`：默认阿里模型名
- `SMARTQA_SOURCE_SYNC_TIMES`：千牛源数据同步时间
- `SMARTQA_DAILY_QC_TIME`：每日 AI 质检时间

| 技术 | 版本/用途 |
| --- | --- |
| SQLAlchemy | `>=2.0.51,<2.1`，ORM、同步/异步数据库连接、模型映射 |

### 2.4 缓存、会话与限流

| 技术 | 用途 |
| --- | --- |
| Redis | 登录会话、系统参数、租户配置、业务缓存 |

### 2.5 认证、权限与多租户

| 技术/模块 | 用途 |
| --- | --- |
| PyJWT | 生成和解析登录 token |

| 自定义中间件 `RequestGuardMiddleware` | IP 黑名单、写保护、请求日志 |

当前权限模型支持老板端和客服端两类业务角色，并通过租户上下文隔离业务数据。

### 2.6 定时任务

| --- | --- |
| APScheduler | 在后端进程内运行定时任务 |

| asyncio.to_thread | 将同步的数据流水线放到线程中执行，避免阻塞事件循环 |

- 定时同步千牛源数据
- 重建 ODS/DIM/DWD 数据

## 3. SmartQA 业务技术链路

SmartQA 业务系统位于 `backend/app/smartqa/`，由后端启动流程显式挂载，核心目标是把千牛客服会话变成可质检、可追踪、可复盘的数据链路。

| 模块 | 用途 |
| --- | --- |
| `sync` | 从千牛源库同步原始聊天和订单/店铺相关数据 |
| `builder` | 将 ODS 原始数据加工成维度表和会话事实表 |
| `conversation` | 会话列表、会话详情、消息明细查询 |
| `qc_rule` | 质检规则和 Prompt 模板管理 |
| `qc_task` | 创建、执行、管理 AI 质检任务 |
| `qc_result` | 保存和查询质检结果、问题证据 |
| `dashboard` | 老板端/客服端统计看板 |
| `staff_user` | 客服账号绑定与权限相关业务 |
| `pipeline.py` | 命令行和定时任务共用的数据流水线 |

数据分层大致如下：

| 数据层 | 表/模型示例 | 用途 |
| --- | --- | --- |
| ODS | `ods_qn_chat_record`、`ods_qn_shop_record` | 保留千牛源数据的关键字段 |
| DIM | `dim_shop`、`dim_product`、`dim_staff`、`dim_customer` | 店铺、商品、客服、客户等维度数据 |
| DWD | `dwd_qn_conversation`、`dwd_qn_message` | 标准化后的会话和消息明细 |
| QC | `qc_task`、`qc_result`、`qc_issue`、`qc_issue_evidence` | AI 质检任务、结果、问题和证据 |
| Log | `model_call_log` | 记录模型请求、响应、token 用量和错误信息 |

## 4. AI 模型调用技术栈

| 技术 | 用途 |
| --- | --- |
| 阿里百炼 DashScope | 提供通义千问 qwen 模型服务 |
| OpenAI Python SDK | 通过 OpenAI 兼容接口调用阿里模型 |
| OpenAI 兼容接口 | 使用 `/compatible-mode/v1/chat/completions` 类似 OpenAI 的请求/响应格式 |
| JSON 模式 | 要求模型输出结构化 JSON，便于后端解析并入库 |

项目没有使用 LangChain。当前调用方式是：

```text
业务代码 -> OpenAI SDK -> 阿里百炼 OpenAI 兼容接口 -> qwen 模型 -> JSON 质检结果
```

相关代码位置：

- `backend/app/smartqa/qc_task/ai_client.py`
- `backend/app/smartqa/pipeline.py`

关键配置示例：

```env
OPENAI_API_KEY=你的阿里百炼 API Key
OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
SMARTQA_ALI_MODEL_NAME=qwen3.7-plus
```

模型请求中主要使用：

- `model`：模型名称，例如 `qwen3.7-plus`
- `messages`：system/user 对话消息
- `temperature`：控制输出随机性
- `max_tokens`：控制最大输出长度
- `response_format={"type": "json_object"}`：要求 JSON 输出

## 8. 项目运行时主要流程

### 8.1 普通接口请求

```text
浏览器页面
  -> Axios
  -> Nginx / Vite 开发代理
  -> FastAPI 路由
  -> 中间件：CORS / 请求守卫 / GZip / Correlation ID / 租户上下文
  -> Service / CRUD
  -> MySQL / Redis
  -> JSON 响应
```

### 8.2 AI 质检流程

```text
千牛源库
  -> SmartQA 同步流水线
  -> ODS 原始表
  -> DIM/DWD 标准化会话数据
  -> 创建 qc_task
  -> 拼接 Prompt
  -> OpenAI SDK 调用阿里百炼 qwen 模型
  -> 解析 JSON
  -> 写入 qc_result / qc_issue / model_call_log
  -> 前端展示质检结果
```

### 8.3 定时任务流程

```text
APScheduler
  -> 按 SMARTQA_SOURCE_SYNC_TIMES 同步千牛源数据
  -> 按 SMARTQA_DAILY_QC_TIME 抽样质检
  -> 根据 SMARTQA_DAILY_QC_EXECUTE 决定是否立即调用模型
```

## 9. 当前项目没有使用的技术

| 技术 | 说明 |
| --- | --- |
| LangChain | 当前没有引入。AI 调用是业务代码直接通过 OpenAI SDK 请求阿里百炼兼容接口 |
| Celery | 当前没有作为任务队列使用。定时任务由 APScheduler 执行 |
| 微服务网关 | 当前是单后端服务，Nginx 做反向代理 |
| 前端 SSR | 当前是 Vite 构建的单页应用 |
